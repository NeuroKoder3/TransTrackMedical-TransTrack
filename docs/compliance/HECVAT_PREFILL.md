## Section 3 — Architecture and Data Flow

| # | Question | Response |
|---|----------|----------|
| 3.01 | Deployment model | **Offline-first desktop** (Electron) running on customer-owned workstations. Data at rest is encrypted SQLite (SQLCipher / better-sqlite3-multiple-ciphers, AES-256). No cloud component required for core operation. Optional server tier is opt-in only. |
| 3.02 | Hosting environment | None for the core product (customer-side desktop). Optional server tier is customer-deployed (Docker containers) — customer chooses cloud provider or on-prem. |
| 3.03 | Data centres / regions | N/A for desktop product. Optional server tier — customer choice. |
| 3.04 | Multi-tenant or single-tenant? | Each desktop installation is isolated to a single customer organisation. The optional server tier supports multiple tenants with hard org_id isolation enforced at every query (cross-org access tested in `tests/cross-org-access.test.cjs`). |
| 3.05 | Sub-processors / third-party services | Core product: none required. Optional server tier may use a customer-chosen identity provider (OIDC/SAML), Epic on FHIR (per customer registration), and customer-chosen SIEM forwarder. |
| 3.06 | Data residency controls | Core product: data physically stored on customer-owned device (full residency control). Server tier: customer-deployed (full residency control). |

## Section 4 — Authentication and Access Control

| # | Question | Response |
|---|----------|----------|
| 4.01 | Authentication mechanisms | Local-account authentication (bcrypt-hashed passwords with configurable complexity), session timeout (15-minute idle), failed-login lockout, password history. Multi-factor authentication (TOTP per RFC 6238 + backup codes) supported and configurable per organisation. |
| 4.02 | Single sign-on supported? | Server tier: SAML 2.0 and OIDC supported via `@node-saml/node-saml` and `openid-client`. Desktop core: not supported in v1.x; roadmap item. |
| 4.03 | Role-based access control | Yes. Roles: admin, coordinator, physician, user, viewer, regulator. Permissions enforced at the IPC layer; cross-role access requires explicit promotion path (audit-logged). |
| 4.04 | Privileged-access workflow | Sensitive actions (encryption key rotation, account unlock, audit log read) require admin role, are audit-logged, and many require a free-text justification ("Access Justification" feature). |
| 4.05 | Audit log immutability | Audit log table has SQL triggers that ABORT any UPDATE or DELETE attempt. Application layer also rejects mutations. Tested in `tests/compliance.test.cjs`. |

## Section 5 — Data Protection

| # | Question | Response |
|---|----------|----------|
| 5.01 | Encryption at rest | AES-256 via SQLCipher. Key managed by `electron/services/encryptionKeyManagement.cjs`; rotation supported and audit-logged. |
| 5.02 | Encryption in transit | All optional-server-tier traffic uses TLS 1.2+. Certificate-pinning available for FHIR/Epic backends in customer config. |
| 5.03 | Key management | Per-installation encryption key derived from machine identifier + user-supplied passphrase. Rotation produces a re-keyed copy of the database; old copy is securely overwritten. |
| 5.04 | Data backup | Customer-controlled. Local backup utility writes encrypted snapshots; restore is also customer-driven. Disaster-recovery procedure documented at `docs/DISASTER_RECOVERY.md`. |
| 5.05 | Secure deletion | On user delete, rows are removed from active tables. Audit-log row records the deletion for the regulatory retention window. Database-file destruction on termination is the customer's responsibility (offline-first model — see BAA Section 5.3). |
| 5.06 | Patch / vulnerability management | npm audit run on every CI build (`npm run audit`). Dependencies pinned. Patch-package preserves any local fixes across upstream releases. Security advisories monitored via GitHub Dependabot. |
